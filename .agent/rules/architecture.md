# Clean Architecture Frontend Guidelines

This document outlines the architectural rules for the frontend development of the Litera Web project using React 19, TypeScript, and Vite.

## Main Directory Structure
The `src/` folder structure must be maintained according to its responsibilities:
- `src/api/`: Contains Axios instance configuration and interceptors (e.g., `apiClient.ts`).
- `src/assets/`: Contains static files like images (SVG, PNG) and global CSS (`index.css`, `App.css`).
- `src/components/`: Reusable React components.
  - `src/components/ui/`: Base UI components generated by Shadcn UI.
  - `src/components/layout/`: Specific components for page layouts (Header, Footer, Sidebar).
- `src/context/`: React Context API for global state if needed (e.g., AuthContext, ThemeContext).
- `src/layouts/`: Page wrapper components (e.g., `AuthLayout`, `DashboardLayout`) that use `<Outlet />` from React Router.
- `src/lib/`: Pure helper utility functions (e.g., date formatting, `utils.ts` for Tailwind).
- `src/pages/`: Page-level components directly mapped to application routes (e.g., `Home.tsx`, `Login.tsx`).
- `src/services/`: Contains API call abstraction functions using the Axios instance.
- `src/types/`: Global TypeScript type definitions (Interfaces, Type Aliases).

## Naming Conventions
- **Components**: Use *PascalCase* for file names and component functions (`Button.tsx`, `UserProfile.tsx`).
- **Custom Hooks**: Use *camelCase* with the `use` prefix (`useAuth.ts`, `useFetchData.ts`).
- **Utility & Service Functions**: Use *camelCase* (`formatDate.ts`, `authService.ts`).
- **Data Types (Type/Interface)**: Use *PascalCase* (optional `I` prefix for interfaces like `IUser` or regular types like `User`).

## State Management
- **Server State**: MUST use `@tanstack/react-query` for fetching, caching, and syncing server data. Do not use `useEffect` + local state for API calls.
- **Client Global State**: Use React Context API limitedly for rarely changing data (such as account themes or login status).
- **Client Local State**: Use `useState` and `useReducer` for specific component state (e.g., modal open, text input).
- **Form State**: Use `react-hook-form` combined with `@hookform/resolvers/zod` for schema validation.

## Routing
- Use `react-router` v7.
- Define routes in a single standard configuration file (e.g., `App.tsx` or `src/routes/index.tsx`).
- Apply *nested routing* for layouts (e.g., root routes wrapped by Navbar/Footer).

## TypeScript
- Never use `any`, ensure every data type is defined.
- Use `interface` for object structure definitions and `type` for union/intersection aliases.
